## Trabajo 4 — Corrección: las marcaciones no se guardaban en el SIA

**Commit:** `(pendiente de commit)` — `fix: formato de fecha ISO en Asistencia para SQL Server 2008`

### Problema
Al importar el CSV, ninguna marcación entraba a la base: todas las filas se
reportaban como «inválidas», incluso con fechas correctas del año en curso.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Models/Sia/Asistencia.php` | Formato de fecha ISO con «T» (`Y-m-d\TH:i:s`). |
| `app/Http/Controllers/MarcacionController.php` | Lector de CSV tolerante (separador `,` o `;`, varios formatos de fecha/hora). |
| `tests/Feature/MarcacionControllerTest.php` | Pruebas del caso Excel y fecha basura. |

### Problema técnico y solución
El SQL Server 2008 del SIA, con el idioma en español, leía la fecha
`2026-07-16` como año-día-mes (día 07, mes 16) y la rechazaba por «fuera de
intervalo». Se aplicó el mismo formato de fecha ISO con «T» que ya usaba el
registro de funcionarios (único que el servidor interpreta igual en cualquier
idioma). Además, el lector de CSV pasó a tolerar archivos reguardados desde
Excel (separador punto y coma, hora con o sin segundos, fecha en otros
formatos).

### Verificación
```bash
php artisan test --compact --filter=MarcacionControllerTest   # 16 pruebas en verde
```
Prueba contra la base real: la marcación se inserta correctamente (fila de
prueba insertada y luego borrada).

## Trabajo 5 — Exportar por rango con modal y envío directo a la base

**Commit:** `(pendiente de commit)` — `feat: enviar marcaciones del equipo directo al SIA por rango`

### Problema
La descarga de CSV era de todo el historial y obligaba a pasar por una
pantalla lenta. Se pidió elegir el rango de fechas en una ventana y, además
de descargar, poder enviar las marcaciones directo a la base del SIA sin
descargar ni reimportar.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Services/RegistroAsistenciaSia.php` | **Nuevo.** Servicio compartido: registra marcaciones en el SIA (cruce por funcionario, sin duplicar, descarta fecha basura). |
| `app/Http/Controllers/MarcacionController.php` | La importación de CSV usa el servicio compartido. |
| `app/Http/Controllers/EquipoController.php` | Filtro por rango en la exportación + envío directo a la base (`sincronizarMarcaciones`). |
| `resources/views/equipos/index.blade.php` | Ventana de rango con dos botones: «Descargar CSV» y «Enviar a la BD». |
| `resources/views/layouts/app.blade.php` | Estilos de la ventana (modal). |
| `routes/web.php` | Ruta de envío directo. |
| `tests/Feature/EquipoControllerTest.php` | Pruebas del filtro por rango y del envío directo. |

### Solución
Se creó un servicio único que registra marcaciones en el SIA con las mismas
reglas, sin importar la fuente (archivo CSV o lectura directa del equipo). En
el listado de Equipos, «Exportar marcaciones» abre una ventana donde se elige
el rango de fechas una sola vez y sirve para las dos acciones: descargar el
CSV de ese rango o enviarlo directo a la base del SIA (con confirmación,
porque escribe en producción). Se eliminó la duplicación de lógica entre la
importación y el envío directo.

### Verificación


**Informe de trabajo — 21/07/2026**
**Responsable:** nachoDev

**Actividad:** Se conectó el registro de marcaciones de los equipos
biométricos con la base de datos del SIA, en ambos sentidos, y se corrigió el
defecto que impedía guardarlas.

**Problema detectado:**
Las marcaciones de los equipos biométricos no se podían llevar a la base del
SIA: no había forma de exportarlas por rango de fechas ni de cargarlas, y al
intentar guardarlas la base las rechazaba por un problema de formato de fecha.

**Solución aplicada:**
Se habilitó la exportación de las marcaciones de cada equipo a un archivo por
rango de fechas y su carga a la base del SIA. Además se agregó el envío
directo a la base (sin pasar por el archivo), eligiendo el rango en una
ventana. Se corrigió el defecto que impedía guardar las marcaciones: la base
del SIA (en español) interpretaba mal las fechas; se aplicó el formato
correcto y ahora se guardan sin problemas. El sistema evita duplicar
marcaciones ya cargadas y descarta automáticamente las fechas inválidas que
arrastran los relojes con la batería agotada.

**Detalle de lo entregado:**
1. Exportación de las marcaciones de cada equipo a archivo CSV por rango de
   fechas.
2. Carga de ese archivo a la base del SIA, sin duplicar y descartando fechas
   inválidas del reloj.
3. Envío directo de las marcaciones del equipo a la base del SIA por rango,
   sin necesidad de descargar y volver a cargar.
4. Corrección del defecto que impedía guardar marcaciones en la base del SIA.
5. Lector de archivos tolerante (soporta archivos reguardados desde Excel).
6. Reordenamiento del menú lateral según el uso diario.

**Resultado:** El sistema ya lleva las marcaciones de los equipos biométricos
a la base del SIA de forma confiable; 95 pruebas automáticas en verde.

**Nota:** Las marcaciones que el reloj registra con fecha equivocada (por la
batería interna agotada) se descartan a propósito para no ensuciar la base. La
solución de fondo es corregir la fecha/hora en el equipo físico.
