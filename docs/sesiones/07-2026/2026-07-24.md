## Trabajo 1 — Nueva base MySQL local (modelos + migraciones espejo del SIA)

**Commit:** `d22a82b` — `nuevo sistema`

### Problema
El sistema dependía por completo del SQL Server legado (SIA) para leer personas,
marcaciones, profesiones y horarios. Se necesitaba que SISBIO fuera dueño de sus
datos en su propia base MySQL, replicando los campos del SIA tal cual (mismos
nombres, longitudes y nulabilidad).

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Models/Persona.php` | Modelo local (tabla `personas`, `IdPersona`→`ci`), SoftDeletes, `$fillable` camelCase, relaciones. |
| `app/Models/Asistencia.php` | Modelo local de marcaciones (tabla `asistencias`, col `ci`). |
| `app/Models/Profesion.php`, `app/Models/Turno.php` | Modelos locales de catálogos (`profesiones`, `turnos`). |
| `database/migrations/*_create_personas_table.php` | Tabla espejo: id Laravel + timestamps + softDeletes; los 16 campos del SIA idénticos; `ci` `unique()`. |
| `database/migrations/*_create_{asistencias,profesions,turnos}_table.php` | Tablas espejo de marcaciones, profesiones y turnos. |

### Solución
Se crearon las tablas MySQL locales replicando el esquema del SIA (misma
identidad de campos, `char()` para longitud fija) pero con la convención del
sistema: id autoincremental de Laravel, timestamps y borrado lógico
(softDeletes). La clave del SIA (`IdPersona`) pasa de PK a columna `unique()` y se
renombra a `ci`; las tablas y columnas quedan en convención local (tablas en
minúscula plural, columnas camelCase). Los modelos legados `App\Models\Sia\*`
(conexión `sqlsrv`, solo lectura) siguen existiendo para las piezas aún no
migradas.

## Trabajo 2 — Comandos de copia SIA → MySQL (tabla por tabla)

**Commit:** `d22a82b` · `9828552` · `a39e6d8` · `681d04c` · `c9a2fc2` — `nuevo sistema`

### Problema
Había que trasladar los datos reales del SQL Server a MySQL sin tocar el origen
(sigue solo-lectura), de forma repetible y sin duplicar filas al reejecutar.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Console/Commands/MigrarPersonasSia.php` | `sia:migrar-personas` — copia Personas→`personas`. |
| `app/Console/Commands/MigrarMarcacionesSia.php` | `sia:migrar-marcaciones` — Asistencia→`asistencias` (~4.4M filas). |
| `app/Console/Commands/MigrarProfesionesSia.php` | `sia:migrar-profesiones` — catálogo. |
| `app/Console/Commands/MigrarHorariosSia.php` | `sia:migrar-horarios` — DiaTurnos→`turnos`. |
| `app/Console/Commands/MigrarLicenciasSia.php` | `sia:migrar-licencias` — Licencias→`licencias` (+ FK `turno_id`). |
| `app/Console/Commands/MigrarAsignacionTurnosSia.php` | `sia:migrar-asignacion-turnos` (+ FK `turno_id`). |
| `app/Console/Commands/MigrarDiasExcepcionalesSia.php` | `sia:migrar-dias-excepcionales` — Calendario→`dias_excepcionales`. |
| `app/Models/{Licencia,AsignacionTurno,DiaExcepcional}.php` | Modelos locales de las tablas nuevas. |
| `database/migrations/*_create_{licencias,asignacion_turnos,dia_excepcionals}_table.php` | Tablas espejo nuevas. |

### Solución
Siete comandos, todos **idempotentes** (upsert por la clave del SIA): recortan el
padding de los `char()`, pueblan `created_at`/`updated_at` y **no tocan el
origen**. Cada comando mapea SIA→local con un array `MAPA` const (PascalCase del
SQL Server → camelCase local). Las tablas grandes (personas, marcaciones,
licencias, asignacion-turnos, dias-excepcionales) usan **`cursor()`** en vez de
`chunk()`: en SQL Server 2008 el `chunk()` pagina con `ROW_NUMBER` (O(n²), parece
colgado); los catálogos chicos (profesiones, horarios) siguen con `chunk()`.
`licencias` y `asignacion_turnos` además llevan una FK **`turno_id`→`turnos.id`**
que el comando resuelve cruzando el `idTurno` contra la tabla `turnos` local (por
eso `turnos` debe migrarse antes).


## Trabajo 3 — Seeder maestro que orquesta toda la migración

**Commit:** `a39e6d8` · `c9a2fc2` · `681d04c` — `nuevo sistema`

### Problema
Correr la migración a mano era propenso a errores de orden (las FK exigen migrar
`turnos` antes que licencias/asignaciones) y no dejaba el sistema en un estado
limpio y reproducible.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `database/seeders/MigrarSiaSeeder.php` | Orquestador: `migrate:fresh --seed` + los 7 comandos de copia en orden. |

### Solución
`php artisan db:seed --class=MigrarSiaSeeder` primero corre `migrate:fresh --seed`
(borra todo, recrea el esquema y el usuario/roles) y luego ejecuta los **7**
comandos de copia en el orden correcto (profesiones, personas, horarios,
marcaciones, licencias, asignacion-turnos, dias-excepcionales — horarios antes de
licencias/asignaciones por la FK). En entorno de pruebas **no** hace
`migrate:fresh` (rompería `RefreshDatabase`). No está en `DatabaseSeeder`: se
corre aparte, a propósito.

> **Entornos:** el código se edita en Linux; la migración real la corre el usuario
> en Windows (con `pdo_sqlsrv` y red al SIA). Son checkouts distintos → hay que
> commit + push + pull.

## Trabajo 4 — Aislamiento de las pruebas frente al SQL Server

**Commit:** `d22a82b` · `9828552` · `a39e6d8` · `681d04c` — `nuevo sistema`

### Problema
Con `pdo_sqlsrv` instalado, cualquier test que renderizara el tablero sin falsear
la conexión `sia` pegaba al **SQL Server real** y se colgaba por login timeout. El
suite tardaba ~124 s en vez de ~3 s.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `tests/Pest.php` | `beforeEach` global que apunta `sia`→sqlite vacío en todos los tests Feature, y helper `fakeSiaDatabase()` que crea las tablas del SIA falso. |
| `tests/Feature/Migrar*SiaTest.php` | Un test por comando de copia (siembran el SIA falso y verifican el upsert). |

### Solución
Se añadió un `beforeEach` global en `tests/Pest.php` que redirige la conexión
`sia` a un sqlite en memoria para todos los tests, y un helper `fakeSiaDatabase()`
que crea las tablas del origen falso. Así ningún test toca el SQL Server real y el
suite vuelve a correr en segundos. Al agregar una tabla nueva a la migración hay
que sumar su tabla a `fakeSiaDatabase()` y su comando a `MigrarSiaSeeder`.

## Trabajo 5 — «Flip»: la aplicación lee de MySQL local

**Commit:** `2a3ddd3` — `nuevo sistema migrado`

### Problema
Aunque los datos ya estaban en MySQL, la aplicación seguía **leyéndolos del SIA**
(SQL Server) en las pantallas de Marcaciones, Horarios, Funcionarios y Reportes.
Había que cambiarla para que consumiera las tablas locales.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Http/Controllers/MarcacionController.php` | Lee `App\Models\Asistencia` local (cols `fecha`/`hora`/`tipo`). |
| `app/Http/Controllers/DiaTurnoController.php` | CRUD de horarios lee/escribe `App\Models\Turno` local. |
| `app/Http/Controllers/PersonaController.php` | Funcionarios sobre `App\Models\Persona` local (URLs por `ci`). |
| `app/Http/Controllers/ReporteMarcacionController.php` | Reporte lee Persona/Asistencia/Profesion locales. |
| `app/Policies/{Asistencia,Turno,Persona}Policy.php` | Policies nuevas para los modelos locales (mismos permisos). |
| `app/Http/Requests/{Store,Update}PersonaRequest.php` | Reglas en camelCase (`unique('personas','ci')`, `exists('profesiones','codigoProfesion')`). |
| `database/factories/{Persona,Asistencia,Profesion,Turno}Factory.php` | Factories locales (camelCase). |
| `resources/views/{funcionarios,horarios,marcaciones,reportes}/**` | Vistas migradas a camelCase (`CorreoE`→`correo`, etc.). |
| `tests/Feature/{Marcacion,DiaTurno,Persona,ReporteMarcacion}ControllerTest.php` | Los tests de display siembran tablas locales; los de import siguen sembrando el SIA. |

### Solución
Se apuntaron los controllers de Marcaciones, Horarios, Funcionarios y Reportes a
los modelos MySQL locales, con sus policies, factories y requests nuevos. `Persona`
usa `getRouteKeyName()='ci'` (URLs por carné). Las vistas pasaron a camelCase.
Queda **pendiente** el flip del `DashboardController` (aún usa `Sia\*`) y del
**import** de CSV / `EquipoController::sincronizarMarcaciones`, que todavía
escriben al SIA vía `RegistroAsistenciaSia`; hasta migrarlos, lo importado no
aparece en el listado local hasta re-migrar.

### Verificación
```bash
php artisan test --compact --filter="MarcacionControllerTest|DiaTurnoControllerTest|PersonaControllerTest|ReporteMarcacionControllerTest"
```

**Actividad:** Se trasladó la información del sistema antiguo (SIA) a la base de
datos propia del nuevo sistema, y se empezó a hacer que el sistema use su propia

**Problema detectado:**
El sistema dependía por completo de la base de datos del sistema antiguo para
mostrar funcionarios, marcaciones, horarios y demás. Eso lo ataba a un servidor
viejo que se quiere retirar.

**Solución aplicada:**
1. **Base de datos propia:** se creó en el nuevo sistema una copia de las tablas

   asignación de turnos y días especiales), respetando todos los datos.
2. **Traslado de datos:** se prepararon procesos que copian la información del
   sistema antiguo a la nueva base sin modificar el original, y que se pueden
   volver a correr sin duplicar nada.
3. **Un solo botón de migración:** se dejó un proceso que arma toda la base nueva
   y trae los datos en el orden correcto, listo para ejecutarse cuando se conecte
   al servidor antiguo.
4. **El sistema empieza a usar su base:** las pantallas de Marcaciones, Horarios,
   Funcionarios y Reportes ya leen de la base nueva; quedan por pasar el tablero
   de inicio y la importación de marcaciones.

**Resultado:** El sistema ya tiene su propia base de datos con la información del
sistema antiguo y varias pantallas funcionando sobre ella. Las 155 pruebas
automáticas del sistema quedan en verde. El objetivo final es dejar de usar el
servidor antiguo por completo.

> Nota técnica: el traslado real de los datos se ejecuta desde el equipo Windows
> que tiene conexión al servidor antiguo (SQL Server).
