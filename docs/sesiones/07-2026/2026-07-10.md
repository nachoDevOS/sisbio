## Trabajo 1 — Roles y permisos (Filament Shield + Spatie)

**Commit:** `(pendiente de commit)` — `feat: roles y permisos con filament-shield`

### Problema
El panel no tenía control de acceso: cualquier usuario autenticado podía hacer
todo. Se necesitaban roles y permisos administrables.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `composer.json` | Dependencias `bezhansalleh/filament-shield` y `spatie/laravel-permission`. |
| `app/Providers/Filament/AdminPanelProvider.php` | Registro del plugin de Shield. |
| `app/Models/User.php` | Trait `HasRoles`. |
| `config/permission.php`, `database/migrations/*_create_permission_tables.php` | Tablas de permisos. |
| `app/Policies/{EquipoPolicy,RolePolicy}.php` | Policies generadas. |

### Solución
Se instaló Shield + Spatie, se registró el plugin, se agregó `HasRoles` al modelo
`User` y se generaron policies y permisos. Se asignó `super_admin` a
`admin@admin.com`. La gestión de roles queda en `/admin/shield/roles`.

**Después:**
```php
// app/Models/User.php
use Spatie\Permission\Traits\HasRoles;

class User extends Authenticatable
{
    use HasFactory, HasRoles, Notifiable;
}
```

## Trabajo 2 — Recorte de permisos a 5 por recurso

**Commit:** `(pendiente de commit)` — `chore: limitar permisos a ver/crear/editar/eliminar`

### Problema
Shield generaba 12 métodos por recurso (soft-delete, replicate, reorder…) que el
sistema no usa. Solo se necesitan 5: ver listado, ver, crear, actualizar, eliminar.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `config/filament-shield.php` | `methods` y `single_parameter_methods` a solo 5; `super_admin` vía gate; widget `EquiposStats` excluido. |
| `app/Policies/{EquipoPolicy,RolePolicy}.php` | Policies regeneradas con 5 métodos. |

### Solución
Se recortó la config a `viewAny/view/create/update/delete`, se regeneraron las
policies y se borraron de la base los 15 permisos sobrantes (incluido
`View:EquiposStats`). Quedaron 10 permisos (5 × Equipo, 5 × Role).

**Antes:**
```php
'methods' => [
    'viewAny', 'view', 'create', 'update', 'delete', 'deleteAny', 'restore',
    'forceDelete', 'forceDeleteAny', 'restoreAny', 'replicate', 'reorder',
],
```

**Después:**
```php
'methods' => [
    'viewAny', 'view', 'create', 'update', 'delete',
],
```

## Trabajo 3 — Recurso de Usuarios

**Commit:** `(pendiente de commit)` — `feat: gestión de usuarios y asignación de roles`

### Problema
No había forma de crear ni administrar usuarios del panel ni de asignarles roles.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Filament/Resources/Users/**` | Recurso `UserResource` (form, tabla, páginas). |
| `app/Policies/UserPolicy.php` | Policy generada con los 5 permisos. |
| `tests/Feature/UserResourceTest.php` | Pruebas del recurso. |
| `tests/Pest.php` | Helper `asSuperAdmin()`. |
| `tests/Feature/{EquipoResourceTest,ProbarConexionActionTest}.php` | Ahora actúan como super admin. |

### Solución
Recurso en `/admin/users`. Formulario con Nombre, Correo (único), Contraseña
(hasheada por el cast del modelo, opcional en edición) y selector múltiple de
Roles. Tabla con roles como badges y acciones crear/editar/borrar. Redirige al
listado tras guardar, igual que Equipos.
